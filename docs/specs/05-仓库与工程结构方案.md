# 05. 仓库与工程结构方案

日期：2026-06-06  
状态：Active  
用途：定义当前仓库结构的真实现状，以及后续工程结构应如何演进

## 1. 当前仓库真实结构

以当前文件系统为准，仓库顶层结构为：

```text
CallCenter/
├── AGENTS.md
├── README.md
├── docs/
├── services/
│   ├── callcenter-server/
│   └── callcenter-web/
├── server/
└── deploy/
```

需要特别说明：

- `services/callcenter-server/` 是当前项目完整后台入口。
- `services/callcenter-web/` 是当前项目前端入口。
- 根目录 `server/` 是历史后端骨架，不再承载新的 CallCenter 业务。
- `docs/specs/` 承接原 CallCenter `doc/specs/` 的规格事实来源职责。
- `third-party/` 不作为当前默认目录保留。

## 2. 当前各目录职责

### `docs/specs/`

只保留执行期总览文档，用于描述：

- 产品范围
- 规模与非功能要求
- 部署与集成约束
- 总体技术方案
- 仓库与工程结构方案

### `services/callcenter-server/`

当前为后端代码主目录，仍保留 RuoYi-Vue-Plus 裁剪后的聚合结构：

- `ruoyi-admin`
- `ruoyi-common`
- `ruoyi-modules`
- `script`

### `services/callcenter-web/`

当前为前端代码主目录，仍是 plus-ui 裁剪后的单应用结构。

### `deploy/`

当前为私有化部署与运维入口目录，包含：

- `compose.yml`
- `compose.app.yml`
- `ops`
- `release/`
- `observability/`

### `server/`

当前仅作为历史工程骨架保留，不作为 CallCenter 后端入口。

## 3. 当前工程问题

虽然顶层结构已经比早期更干净，但当前工程仍然有明显的演进压力：

### 3.1 后端仍以脚手架心智为主

当前后端目录还是：

- `ruoyi-admin`
- `ruoyi-common-*`
- `ruoyi-system`

这更像“底座继承结构”，而不是“呼叫中心领域结构”。

### 3.2 前端仍以单后台心智为主

当前前端仍然是管理后台结构，尚未区分：

- 管理后台
- 坐席工作台
- 主管工作台
- 质检工作台

### 3.3 文档只剩总览，缺少领域级规格载体

当前模块级 `spec.md` 已经被删除，意味着仓库现在缺少“领域模块逐步落地”的中间层文档。

这会带来两个问题：

- 后续新增业务时容易直接写代码
- 领域边界不容易被持续维护

## 4. 后续推荐结构

在不立即大搬家的前提下，建议后续逐步演进为：

### 后端

```text
services/callcenter-server/
├── callcenter-bootstrap/
├── callcenter-shared/
├── callcenter-domain/
├── callcenter-adapter/
├── callcenter-application/
└── script/
```

### 前端

```text
services/callcenter-web/
├── apps/
├── packages/
└── tooling/
```

### 文档

如果后续开始逐模块落地业务，建议重新引入轻量级模块文档目录，但只在真正需要时建立，不再预先铺满空文档。

推荐形态：

```text
docs/
└── domains/
    ├── agent.md
    ├── call.md
    ├── cti.md
    ├── realtime.md
    ├── chat.md
    ├── recording.md
    ├── quality.md
    ├── ai.md
    ├── report.md
    └── integration.md
```

## 5. 当前建议

对当前仓库来说，最合理的做法不是再继续删除总览文档，而是：

1. 保持 `docs/specs/` 极简但准确
2. 让 `services/callcenter-server/` 和 `services/callcenter-web/` 按后续真实业务逐步长出新结构
3. 当某个领域真正开始实现时，再补对应领域文档

## 6. 结论

当前仓库已经从“启动期大而散”进入“执行期收敛态”，但距离最终的企业级工程结构还有一段路。

接下来的重点不是继续堆过程文档，而是：

- 保持现有总览文档准确
- 让新增代码按领域边界落地
- 在真实业务开始实现时，再补最小必要的领域规格文档
