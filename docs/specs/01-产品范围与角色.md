# 01. 产品范围与角色

日期：2026-06-06  
状态：Active  
用途：定义当前项目要解决什么问题、服务哪些角色、覆盖哪些核心业务能力

## 1. 项目定位

CallCenter 是面向客户本地机房私有化部署的企业级呼叫中心 Web 平台。

当前仓库聚焦第一阶段基础平台建设，目标是在既有后台底座上承载后续呼叫中心核心业务，而不是继续做通用后台扩展。

## 2. 交付模式

- 单客户独立部署
- 客户本地机房私有化交付
- 非 SaaS 多租户模式
- 支持后续跨客户复用，但复用方式是“项目模板演进”，不是运行时多租户

## 3. 目标用户角色

第一阶段重点考虑以下角色：

| 角色 | 主要职责 |
|---|---|
| 坐席 | 处理电话接听、销售外呼、文字聊天、客户沟通记录 |
| 主管 | 查看坐席状态、管理任务、监督服务过程 |
| 质检 | 查看通话或聊天记录，进行质检评分 |
| 管理员 | 配置用户、角色、权限、基础业务参数 |

## 4. 第一阶段核心能力

第一阶段目标能力包括：

1. 客服接听
2. 销售外呼
3. 文字聊天
4. 话务条
5. 来电弹屏
6. 真实电话线路 / CTI 接入
7. 录音
8. 人工质检
9. AI 质检 / 语音转写
10. 复杂报表

## 5. 当前仓库与业务实现状态

需要明确区分“业务目标”和“代码现状”：

- 业务目标已经明确到呼叫中心核心领域。
- 当前代码仓库仍以后台底座和系统管理能力为主。
- 呼叫中心核心业务模块尚未在代码中完整落地。

因此，当前阶段的重点不是继续扩写零散页面，而是：

- 稳定总体架构
- 明确领域边界
- 为后续核心模块落地准备工程结构

## 6. 当前非目标

以下内容当前不作为第一阶段必交能力：

- 运行时多租户
- 预测式外呼
- IVR 复杂流程平台
- 移动端 App
- 计费系统
- 多地区多活
- 大规模第三方渠道聚合平台

## 7. 产品约束

- 呼叫中心核心能力不能按普通 CRUD 思路实现。
- 电话、坐席、CTI、录音、质检、报表都应先明确领域模型。
- 外部系统对接必须通过适配层隔离。
- 后续功能拆分以真实业务边界为准，而不是以页面菜单为准。
