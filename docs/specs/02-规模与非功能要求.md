# 02. 规模与非功能要求

日期：2026-06-06  
状态：Active  
用途：定义第一阶段必须满足的规模、实时性和非功能约束

## 1. 已确认规模

| 指标 | 数值 |
|---|---:|
| 坐席总数 | 10,000 |
| 同时在线坐席 | 3,000 |
| 并发通话 | 3,000 |
| 每日电话量 | 10,000 |
| 来电弹屏实时性 | 3 秒内 |
| 语音转写 | 通话结束后异步转写 |
| 报表 | T+1 统计报表 |

## 2. 基本判断

这不是普通后台管理系统规模，而是大型呼叫中心平台规模。

对架构的直接影响包括：

- 需要稳定的实时连接管理
- 需要可靠的 CTI 事件处理链路
- 需要独立的异步任务链路
- 需要把报表与业务主链路分开设计

## 3. 关键性能目标

### 3.1 实时链路

| 场景 | 目标 |
|---|---:|
| 系统处理来电事件 | 1 秒内 |
| 推送到坐席工作台 | 3 秒内 |
| 坐席状态变化同步 | 3 秒内 |

### 3.2 通话链路

以下指标仍需在 CTI 方案明确后补齐：

- 接听响应时间
- 挂断响应时间
- 转接响应时间
- 通话结束后记录落库时限

### 3.3 AI 与异步链路

- 转写模式：异步
- 触发时机：通话结束后
- 必须支持失败重试
- 必须可做成本控制

### 3.4 报表链路

- 第一阶段以 T+1 离线统计为主
- 不把复杂分析直接压在业务库上
- 实时大屏不作为当前默认能力

## 4. 架构性要求

这些要求已经对实现方式形成强约束：

1. 不能只依赖 HTTP 轮询处理实时业务。
2. CTI 回调必须幂等。
3. 坐席状态、通话状态、连接状态必须分层建模。
4. 录音处理、转写、AI 质检、报表必须走异步链路。
5. 报表查询不能与核心业务读写混为一体。
6. 敏感数据必须具备独立存储与权限控制意识。

## 5. 当前仍待补齐的非功能项

以下内容还没有形成最终结论，但后续必须补齐：

- 系统可用性目标
- 数据保留周期
- 行业合规要求
- 录音下载与审计要求
- 脱敏与导出审批要求

## 6. 当前风险

| 风险 | 说明 |
|---|---|
| 范围过大 | 第一阶段目标覆盖电话、聊天、录音、AI、报表，实施压力大 |
| 实时链路复杂 | 3,000 在线坐席与 3 秒弹屏需要专门设计 |
| CTI 依赖高 | 华为相关系统细节未完全明确 |
| AI 成本不确定 | 大规模录音转写会带来明显成本 |
| 数据合规压力 | 录音、手机号、聊天记录、转写文本都属于高敏感数据 |
