# 03. 部署与集成约束

日期：2026-06-06  
状态：Active  
用途：定义当前项目在部署方式、客户环境和外部系统对接上的硬约束

## 1. 已确认约束

| 项目 | 当前结论 |
|---|---|
| 部署位置 | 客户本地机房 |
| 部署方式 | 私有化部署 |
| 租户模式 | 单客户独立部署 |
| 团队主栈 | Java + Vue |
| 部署要求 | 需要集群、方便部署、方便排障 |
| 数据源要求 | 支持多数据源 |
| 可复用要求 | 需要方便后续迁移与复用 |

## 2. 外部系统约束

### 2.1 CTI / 电话系统

当前已确认客户有华为相关系统，但具体产品和接口形式尚未完全落定。

这意味着：

- 业务系统不能直接耦合华为接口
- 必须保留独立 CTI 适配层
- 通话事件必须转换成系统内部标准事件
- 接听、挂断、转接、外呼控制必须通过统一接口对外暴露

### 2.2 CRM / 工单 / 客户资料系统

当前确认需要对接，但尚未明确：

- 哪个系统是主数据源
- 来电弹屏从哪里查询客户信息
- 通话小结写回哪里
- 聊天记录是否需要回写工单
- 同步方式是实时 API 还是离线同步

## 3. 部署约束

由于采用客户本地机房私有化交付，方案必须满足：

- 支持离线或弱公网安装
- 支持统一部署脚本
- 支持标准化日志查看与健康检查
- 支持版本升级、回滚和排障手册演进
- 支持与客户现有内网系统直接对接

## 4. 当前仓库实际部署形态

当前仓库已经落地的部署资产为：

- `deploy/compose.yml`
- `deploy/compose.app.yml`
- `deploy/ops`
- `deploy/release/`
- `deploy/observability/`

也就是说，当前真实交付入口是：

```text
Docker Compose
+ Bash 运维脚本
+ 基础设施与应用分离编排
+ GitHub Actions / SSH 发布骨架
+ 本地可观测性栈
```

不是 Kubernetes，也不是微服务治理平台。

## 5. 当前技术边界

虽然团队有 Spring Cloud + Nacos 经验，但当前仓库和当前方案都没有选择那条路线作为第一阶段主形态。

当前阶段明确采用：

- 模块化单体
- Docker Compose 私有化部署
- 后端可多实例扩展
- 外部系统通过 adapter 隔离

## 6. 后续必须继续确认的事项

以下仍是后续落地前必须补齐的信息：

1. 华为系统的具体产品名与接口类型
2. 客户是否指定数据库品牌和版本
3. 客户是否指定监控、日志、消息队列平台
4. CRM / 工单 / 客户资料系统的边界与责任划分
5. 客户是否已有容器平台，但这不会反向改变第一阶段默认交付形态
