# 04. 总体技术方案

日期：2026-06-06  
状态：Accepted  
用途：定义当前项目第一阶段的总体技术路线、代码组织原则和交付形态

## 1. 当前最终路线

第一阶段采用：

```text
SDD 规范驱动开发
+ RuoYi-Vue-Plus 5.X 裁剪版
+ plus-ui 裁剪版
+ 模块化单体优先
+ Docker Compose 私有化交付
```

不采用：

- 微服务优先路线
- Kubernetes 作为默认交付前提
- 从零手写基础平台

## 2. 为什么这样选

基于当前已确认事实：

- 客户本地机房私有化部署
- 单客户独立部署
- 大规模实时业务约束
- 团队熟悉 Java + Vue
- 仓库当前已完成 RuoYi-Vue-Plus / plus-ui 裁剪

因此，第一阶段的最优先目标是：

- 降低交付复杂度
- 保持架构可演进
- 避免过早拆服务
- 让 AI 和人都能在清晰上下文中持续推进

## 4. 当前仓库对应关系

当前仓库并没有落地呼叫中心业务模块，而是主要保留了平台底座：

- `services/callcenter-server/ruoyi-admin`
- `services/callcenter-server/ruoyi-common/*`
- `services/callcenter-server/ruoyi-modules/ruoyi-system`
- `services/callcenter-web/src/views/system`
- `services/callcenter-web/src/views/monitor`

这意味着：

- 当前代码仍是“底座就绪、业务待落地”状态
- 文档必须承担更强的先行约束作用

## 5. 核心架构原则

1. 第一阶段采用模块化单体，不预设微服务拆分。
2. 外部系统必须走 adapter。
3. 通话、坐席、CTI、实时、聊天、录音、质检、AI、报表不能按普通 CRUD 建模。
4. 后端实例必须无本地状态，支持多实例部署。
5. WebSocket / SSE 等实时能力必须考虑多实例场景。
6. 报表链路与核心业务链路需要分开演进。

## 6. 对当前代码底座的使用原则

当前保留的 `ruoyi-*` 模块仍可作为过渡底座使用，但不应继续让它自然扩张为最终业务结构。

原则是：

- 保留可复用的系统管理与基础设施能力
- 新增呼叫中心业务时优先沿领域边界落地
- 不把新业务长期塞进 `ruoyi-system`
