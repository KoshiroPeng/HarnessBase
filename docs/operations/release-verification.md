# 发布验证与回滚

## 目标

本文档定义发布完成后的验证动作、观察窗口和回滚判定，让部署不止停留在“发上去了”，而是有明确的健康确认闭环。

## 验证阶段

### 第一阶段：技术健康验证

发布完成后立即检查：

- 应用实例是否启动成功。
- `actuator/health` 是否返回健康状态。
- 指标端点是否可抓取。
- 日志中是否出现启动失败、配置缺失或数据库迁移异常。

### 第二阶段：核心指标验证

至少检查以下指标：

- 请求错误率是否异常上升。
- 响应耗时是否显著恶化。
- JVM 内存、线程和 GC 指标是否异常。
- 数据库连接池是否耗尽。

### 第三阶段：业务探针验证

根据变更内容执行最小业务探针，例如：

- 登录是否成功。
- 项目列表是否可查询。
- 任务创建是否可用。
- 计费相关接口是否返回预期结果。

## 观察窗口

不同环境的建议观察窗口：

| 环境 | 最小观察时长 |
| --- | --- |
| dev | 5 分钟 |
| test | 10 分钟 |
| staging | 15 分钟 |
| prod | 30 分钟，重大变更可延长 |

观察窗口内若出现持续错误、核心指标恶化或业务探针失败，应进入回滚判定。

## 回滚触发条件

出现以下任一情况，应立即评估回滚：

- 健康检查持续失败。
- 错误率超过预设阈值。
- 关键业务路径不可用。
- 数据库迁移导致业务异常且无快速修复路径。
- 关键日志显示配置、权限或依赖异常。

## 回滚前检查

执行回滚前必须确认：

- 目标回滚版本明确可用。
- 旧版本与当前数据库结构是否兼容。
- 是否需要同时回退配置或特性开关。
- 是否需要通知相关业务或值班人员。

## 回滚后动作

- 重新执行健康检查和业务探针。
- 记录回滚时间、原因、影响范围和恢复结果。
- 在事后复盘中补齐根因、缺失信号和流程改进项。

## 当前最小落地建议

1. 基于 `actuator/health` 和 `actuator/prometheus` 建立最小验证流程。
2. 为核心接口定义手工或自动化探针。
3. 把回滚条件写入部署流水线或发布清单。
4. 为生产发布建立观察期责任人。

## 当前脚本入口

当前仓库已提供最小验证脚本入口：

- `deploy/release/verify-release.sh`
- `.github/workflows/server-rollback.yml`

脚本行为约定：

- `DRY_RUN=true` 时，只输出验证计划，不访问真实环境。
- `DRY_RUN=false` 且提供 `TARGET_BASE_URL` 时，检查健康检查和指标端点。
- 若健康检查或指标端点不可用，脚本返回非零退出码，用于阻断发布流程。

当前独立回滚入口：

- `.github/workflows/server-rollback.yml`
- `deploy/release/rollback-via-ssh.sh`

## 维护规则

- 新增关键业务能力时，必须补充对应探针。
- 观测指标变化时，必须同步更新本文档中的验证项。
