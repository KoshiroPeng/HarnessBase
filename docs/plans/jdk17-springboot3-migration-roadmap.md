# JDK 17 / Spring Boot 3 迁移路线

## 目标

本文档用于把 ProjectPilot 从历史 `Java 8 / Spring Boot 2.7` 残留状态迁移到新的目标主线：

- JDK 17
- Spring Boot 3.x
- MySQL 8.x
- 模块化单体后端结构
- Vue 3 + TypeScript + Vite 的 Web 主线

## 当前差异

当前文档主线已经切换，后端基线升级也已完成最小验证，但工程层仍可能存在以下差异：

- `server/` 已在 `JDK 17 + Spring Boot 3.3.x` 下通过 `compile`、`test` 和 `verify`，但模块结构仍未收敛到目标形态。
- 代码中仍需继续排查 `javax.*` 残留、历史注释和旧心智描述。
- 持久化、验证、异常处理和配置装配仍可能依赖旧版习惯。
- 发布、脚本和工作流尚未完全对齐新基线。

## 迁移原则

1. 先统一基线认知，再迁移代码。
2. 先迁移构建与依赖，再迁移代码细节。
3. 先保证可编译、可启动、可验证，再追求结构优化。
4. 不把一次基线迁移扩张成无边界重写。
5. 所有关键变化都要有文档、验证和回滚说明。

## 推荐阶段

### 第一阶段：基线冻结与清单盘点

- 盘点 `server/` 当前依赖、插件、JDK、数据库、脚本和工作流。
- 识别所有 `Java 8 / Boot 2.7 / javax.*` 残留点。
- 确认哪些属于必迁项，哪些属于后续结构优化项。

阶段产出：

- 迁移项清单
- 风险清单
- 验证清单

### 第二阶段：构建与依赖升级

- 切换 JDK 到 17。
- 切换 Spring Boot 到 3.x。
- 升级 Maven 插件与测试插件。
- 校准 MyBatis-Plus、SpringDoc、Lombok、Flyway 等与 Boot 3 对齐的依赖。

阶段目标：

- 先让项目在新基线下可编译。
- 明确哪些三方依赖需要替换或升级。

当前状态补充：

- 本阶段已完成最小闭环验证，`server/` 在 `JDK 17 + Spring Boot 3.3.13` 下 `compile`、`test`、`verify` 已通过。

### 第三阶段：代码语义迁移

- 处理 `javax.* -> jakarta.*`。
- 校准 Validation、Servlet、Filter、Interceptor 等相关代码。
- 检查异常处理、配置属性绑定、序列化与时间类型兼容性。
- 校准测试代码和测试启动方式。

阶段目标：

- 先让项目在新基线下可启动、可跑测试。

### 第四阶段：运行时与数据库对齐

- 确认 MySQL 8.x 的运行与迁移策略。
- 校验 Flyway migration 的执行顺序与兼容性。
- 审查索引、排序、分页、时间字段、字符集与连接参数。
- 更新环境变量模板、部署脚本和运行说明。

阶段目标：

- 让本地、测试和目标环境都围绕新基线运行。

### 第五阶段：结构收敛

- 将核心业务逐步从历史脚手架聚合方式收敛到模块化单体目标结构。
- 按模块拆出 `domain / application / adapter`。
- 将外部系统调用逐步统一到 adapter 或 `ApiClient`。

阶段目标：

- 避免“只是版本升级了，但结构仍继续扩散旧心智”。

### 第六阶段：联调与交付闭环

- 对齐 API 文档、错误码、前端接口约束。
- 更新发布、回滚、观测材料。
- 形成一套可重复执行的迁移后验证清单。

阶段目标：

- 让升级结果真正进入研发、测试和发布主链路。

## 高风险点

- `javax.* -> jakarta.*` 引起的编译或运行时变化。
- Spring Boot 3 下配置装配、验证和异常处理行为变化。
- MySQL 8.x 下 SQL、排序规则、索引和时间字段行为差异。
- 第三方依赖与 Boot 3 的兼容性问题。
- 历史脚本、工作流和环境变量仍假设旧运行时。

## 验收清单

- 后端已在 JDK 17 下编译通过。
- Spring Boot 3.x 应用可以正常启动。
- 关键测试可通过，回归测试未明显退化。
- Flyway migration 可在目标数据库上顺序执行。
- API 文档、错误码、发布文档、观测文档已同步。
- 新增或迁移后的代码不再继续扩张旧基线依赖。

当前已验证事实：

- `server/` 已完成 `compile`、`test`、`verify`。
- Spring Boot 启动测试已在 `JDK 17` 环境下通过。

## 与 Harness Engineering 的关系

这份路线图的价值在于把“升级”从一句口头要求变成可导航、可验证、可收敛的工程闭环：

- 开发知道先改什么。
- 评审知道重点查什么。
- 测试知道应该验证什么。
- 文档知道哪些入口必须同步更新。
