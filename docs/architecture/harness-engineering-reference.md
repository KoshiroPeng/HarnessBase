# Harness Engineering 参考手册

## 目标

本文档面向 ProjectPilot 的开发者、评审者和 AI 协作代理，提供一份可以直接落地使用的 Harness Engineering 参考手册。

读完本文后，读者应能够完成两件事：

1. 判断一个方案、文档、工具链或协作流程，是否真正属于 Harness Engineering。
2. 识别当前 Codex + `AGENTS.md` 工作环境里，哪些部分已经在承担 Harness Engineering 的职责，哪些部分仍需补齐。

## 一句话定义

Harness Engineering 不是“把 prompt 写得更花”，也不是“先做一套平台再做产品”。

它的核心是：

> 用一套可运行、可观察、可验证、可约束的工程支架，把模型能力稳定转化为任务完成能力。

如果只看模型，看到的是“它会不会回答”。
如果引入 Harness Engineering，关心的就是“它能不能在真实环境里持续把事情做对，并且做完后可以被验证”。

## 先区分两个概念

### Harness 是什么

`harness` 是运行时支架，是把模型接入真实工程环境的那一层系统。

它通常负责：

- 任务输入如何组织
- 文档和上下文如何选择
- 工具如何暴露给模型
- 状态和记忆如何保留
- 结果如何验证
- 失败如何归因
- 人如何介入和修正

### Harness Engineering 是什么

`harness-engineering` 是围绕这套支架开展的系统化工程工作。

它关注的不是某一次回答，而是下面这些长期问题：

- 这个协作系统是否容易偏航
- 偏航后能否尽快被发现
- 发现问题后能否定位原因
- 修复后能否用统一标准再次验证
- 一次任务的经验能否变成下一次任务的可复用资产

一句话概括：

- `harness` 是系统本体
- `harness-engineering` 是建设和迭代这套系统的方法论

## 为什么它会变重要

真实软件任务和普通问答完全不是一类问题。

软件任务往往具有这些特征：

- 持续时间长
- 上下文多且会变化
- 会跨文件、跨模块、跨工具
- 经常需要执行、验证、回滚和复查
- 任务完成标准不能只靠“看起来像对的”

如果没有 harness，模型很容易出现这些问题：

- 上下文过长后开始漂移
- 会调用工具，但不知道何时停止
- 会改代码，但无法证明修改没有引入回归
- 只完成表面答案，没有闭环验证
- 人和 AI 的协作约束分散在聊天里，无法复用

Harness Engineering 的价值就在这里：

它把“会生成内容”升级成“能稳定推进任务”。

## 推荐的理解框架

理解 Harness Engineering 时，建议使用下面这个最小模型：

```text
用户意图
   ->
任务规格
   ->
上下文选择
   ->
模型推理
   ->
工具执行
   ->
结果验证
   ->
日志 / 轨迹 / 反馈
   ->
下一轮迭代
```

在这个链路里，模型只是中间一环。

真正决定系统是否可用的，是下面两个问题：

1. 模型每一步看到的上下文，是否足够真实、足够新鲜、足够相关。
2. 模型每一步做出的动作，是否有清晰边界，并且能被后续验证和复盘。

## 一个成熟 Harness 常见的能力模块

下面这些能力不一定要一次性做全，但它们构成了 Harness Engineering 的主要观察维度。

### 1. 任务规格化

把“用户大致想做什么”，转成“本轮需要完成什么、做到什么程度才算完成”的可执行目标。

典型表现：

- 明确任务主线和非目标
- 明确约束条件
- 明确交付物和验收标准

### 2. 上下文选择

决定模型在当前步骤应该读取哪些文档、哪些代码、哪些历史约束。

典型表现：

- 有统一入口而不是靠记忆乱找
- 能按任务场景快速定位资料
- 能避免把无关上下文一股脑塞给模型

### 3. 工具接入

把文件、终端、浏览器、测试器、构建系统、外部服务等能力暴露给模型。

典型表现：

- 工具边界清晰
- 工具使用成本低
- 工具结果能反馈回后续推理

### 4. 项目记忆

把一次次任务里稳定不变的规则、架构、术语和决策沉淀下来，避免每次都重新口头解释。

典型表现：

- 统一规则入口
- 文档导航
- 架构、规范、计划、评审清单等长期资产

### 5. 状态管理

记录当前任务进行到哪里、已经验证了什么、还有什么没完成。

典型表现：

- 有可见的计划或阶段性目标
- 有中间产物
- 有完成前检查

### 6. 验证与评估

判断“做完了没有”和“做对了没有”。

典型表现：

- 测试
- 构建
- 静态检查
- 检查清单
- 需求和设计对照

### 7. 可观测性与轨迹

让人和系统都能看见 agent 做了什么、为什么这样做、哪里失败了。

典型表现：

- 日志
- 命令输出
- 变更记录
- 评审记录
- 运行指标

### 8. 权限和护栏

限制模型能做什么、不能做什么，减少高代价错误。

典型表现：

- 禁止危险命令
- 规定必须遵守的编码和提交流程
- 规定哪些变更必须同步更新文档

### 9. 人机协作接口

定义人何时介入、看什么、如何复核。

典型表现：

- 开始前的任务澄清
- 中途的进度通报
- 最终的验证说明
- 评审模板

## Harness Engineering 和其他概念的区别

### 它不等于 Prompt Engineering

Prompt Engineering 关心“怎么提问更有效”。
Harness Engineering 关心“除了提问之外，整个系统如何让任务更容易成功”。

### 它不等于 Agent Framework

框架只是实现基础设施。
Harness Engineering 更偏工程实践，强调规则、上下文、验证、日志、文档和协作流程如何联合工作。

### 它不等于平台化建设

平台化建设可能是 Harness Engineering 的结果之一，但不是它的前提。

尤其在 MVP 阶段，最好的 Harness Engineering 往往不是先造大平台，而是先搭好最小闭环：

- 统一入口
- 清晰规则
- 最小验证
- 可复用文档

## 面向用户与 AI 协作开发的实践清单

下面这份清单适合在日常开发、评审和任务协作中反复使用。

### 开始任务前

- 先确认当前任务属于产品功能、工程治理还是运行支撑。
- 先确认主线目标和非目标，避免一上来把范围做大。
- 先找到统一入口文档，而不是直接在仓库里散搜。
- 先确认本轮验收标准是什么，避免最后只能凭感觉判断。

### 执行任务时

- 优先读取和当前任务直接相关的文档、代码和约束。
- 使用统一工具链，不临时绕开既有抽象。
- 把中途发现的稳定规则沉淀到文档，而不是留在聊天记录里。
- 每次做出关键实现选择时，都要能说明“为什么这样做更符合当前项目主线”。

### 完成任务前

- 先验证功能是否完成，再验证是否符合约束。
- 对照检查清单确认没有遗漏同步文档、测试和回归验证。
- 用可重复的方式说明结果，不只写主观总结。
- 把可以复用的路径接入导航，避免知识成为孤岛。

## 常见误区

### 误区 1：把 Harness Engineering 理解成“再造一个平台”

这通常会带来大量概念、目录和流程，却不能直接帮助当前任务闭环。

更好的做法是：

- 先让当前主线更稳
- 再把高频问题抽成支架

### 误区 2：把它理解成“写更多文档”

文档只有在承担导航、约束、验证或沉淀作用时，才属于高价值 harness。

如果文档不能帮助人或 AI 更快做出正确动作，它就只是额外维护成本。

### 误区 3：只强化输入，不强化验证

很多团队已经会写长 prompt，却没有统一验证出口。

这会导致系统看起来很聪明，但一到复杂任务就无法稳定结束。

### 误区 4：只有 AI 工具链，没有人机协作协议

AI 不会天然知道哪些风险需要停下来、哪些决定需要升级确认。

如果没有人机协作规则，再强的模型也容易在错误方向上高效率前进。

## ProjectPilot 当前环境中，哪些部分已经属于 Harness Engineering

这一节专门回答“结合当前 Codex / `AGENTS.md` 工作方式，哪些部分已经是 harness”的问题。

### 1. `AGENTS.md` 承担了统一规则入口和硬护栏

`AGENTS.md` 已经不是普通说明文件，而是当前仓库最核心的 harness 资产之一。

它已经承担了这些职责：

- 统一语言和编码要求
- 固定技术栈基线
- 固定分层依赖方向
- 约束外部调用方式
- 约束注入方式、日志方式和代码规模
- 约束测试和回归要求
- 约束 Git 提交流程

这属于 Harness Engineering 里的：

- 任务约束
- 权限和护栏
- 可验证规范

### 2. `docs/README.md` 承担了上下文选择入口

很多 agent 失败，不是因为不会写代码，而是拿错上下文。

`docs/README.md` 当前已经把文档按“我现在要做什么”组织起来，这正是典型的 harness 设计。

它解决的是：

- 人和 AI 从哪里开始读
- 不同任务优先读什么
- 如何避免把架构、规范、评审、发布材料混成一团

这属于 Harness Engineering 里的：

- 上下文选择
- 项目记忆
- 导航层

### 3. `docs/architecture/`、`docs/design/`、`docs/plans/`、`docs/reference/` 共同承担项目记忆

这些目录不是“补充材料”，而是任务执行时的长期记忆层。

它们分别承担：

- `docs/architecture/`：系统边界、依赖方向、数据流、Harness 纠偏口径
- `docs/design/`：产品能力、业务场景和 Web MVP 主线
- `docs/plans/`：迭代节奏、当前优先级和执行顺序
- `docs/reference/`：API、错误码等容易在实现中反复查询的事实性信息

这属于 Harness Engineering 里的：

- 项目记忆
- 任务规格化
- 事实参照层

### 4. `docs/reviews/` 和 `docs/conventions/` 承担验证器角色

很多团队把评审清单当成“事后检查表”，但在 agent 场景下，它们本质上就是外置 evaluator。

当前仓库里：

- `docs/conventions/` 提供编码、命名、错误处理、测试等规范
- `docs/reviews/` 提供需求、设计、代码、测试等阶段的评审清单和模板

它们让“好不好”不再只取决于模型主观判断，而是有了相对稳定的外部评估标准。

这属于 Harness Engineering 里的：

- 验证与评估
- 人机协作接口
- 风险收敛机制

### 5. `deploy/release/` 和 `deploy/observability/` 承担运行支撑与可观测性

这些内容虽然不是当前产品主线，但它们仍然是重要 harness 资产。

它们已经覆盖了：

- 发布前检查
- 环境变量模板
- 回滚材料
- 本地可观测性方案

这意味着当前仓库并不只是“能改代码”，而是已经部分具备“能验证运行状态、能准备上线支撑”的能力。

这属于 Harness Engineering 里的：

- 运行环境支架
- 可观测性
- 发布验证

### 6. Codex 的工具能力构成了运行时执行层

当前这个 Codex 工作环境，本身也是 harness 的一部分。

从运行方式看，它已经提供了：

- 文件读取和写入能力
- Shell 执行能力
- Web 检索能力
- 计划更新能力
- 文档与代码联动修改能力

这意味着模型不是在“空聊”，而是在一个具备执行、检查和反馈能力的运行壳里工作。

这属于 Harness Engineering 里的：

- 工具接入
- 执行层
- 反馈闭环

### 7. Codex 的协作协议构成了人机交互层

当前工作方式里还有一层非常关键，但容易被忽略的 harness：协作协议。

例如：

- 开始前先说明要做什么
- 过程中持续汇报进度
- 修改前先解释改动方向
- 完成后交代验证结果和限制

这让用户始终知道 agent 在做什么，也让 agent 不能偷偷跳过关键步骤。

这属于 Harness Engineering 里的：

- 人机协作接口
- 干预机制
- 轨迹透明化

## 结合当前环境，可以怎样理解“已经有了哪些 Harness”

如果把当前环境抽象成一张图，可以这样理解：

```text
用户目标
   ->
AGENTS.md 统一规则
   ->
docs/README.md 场景导航
   ->
architecture / design / plans / reference 提供上下文
   ->
Codex 读取文档、搜索代码、调用工具
   ->
conventions / reviews / tests 进行验证
   ->
deploy / observability 提供运行支撑
   ->
最终产出代码、文档和验证说明
```

这张图说明一个事实：

ProjectPilot 当前已经不只是“一个代码仓库”，而是一个带有初步 harness 结构的 AI 协作工程环境。

## 当前环境还缺什么

虽然基础已经有了，但如果从更成熟的 Harness Engineering 角度看，当前仓库仍有几类能力可以继续增强。

### 1. 已补齐统一的任务状态模板，但还需要持续使用

当前仓库已经补齐统一的任务状态模板：

- [docs/plans/task-status-template.md](../plans/task-status-template.md)

它已经可以用于统一记录：

- 任务做到哪了
- 哪些文档已读、已更、待更
- 哪些验证已完成、哪些还未完成
- 当前还有哪些风险与阻塞

下一步的重点不再是“有没有状态模板”，而是让功能开发、缺陷修复和发布支撑任务都真正使用它。

### 2. 已补齐统一的验证证据模板，但还需要持续使用

当前仓库已经补齐了统一的验证证据输出模板：

- [docs/reviews/templates/verification-evidence-template.md](../reviews/templates/verification-evidence-template.md)

它已经可以用于记录：

- 改了什么
- 用什么命令或操作验证
- 结果是什么
- 还有哪些未覆盖风险

下一步的重点不再是“有没有模板”，而是让开发、自检、测试验证和发布验证阶段都真正持续使用它。

### 3. 已补齐基础的 AI 协作场景分流，但仍可继续细化

当前仓库已经补齐了基础场景分流入口：

- [docs/README.md](../README.md) 负责总导航
- [docs/conventions/task-startup-checklist.md](../conventions/task-startup-checklist.md) 负责任务启动路径
- [docs/plans/task-status-template.md](../plans/task-status-template.md) 负责任务状态沉淀
- [docs/reviews/templates/verification-evidence-template.md](../reviews/templates/verification-evidence-template.md) 负责验证结果沉淀

它们已经可以覆盖：

- 功能开发时优先读什么
- 需求、设计、代码、测试和发布任务分别该读什么
- 执行中如何记录状态
- 完成后如何沉淀验证证据

后续仍可继续细化的方向包括：

- 修复线上问题时的专用阅读顺序
- 新增接口或迁移脚本时的强制联读路径
- 仅做文档整理时的最小阅读集合

### 4. 缺少更自动化的验证闭环

当前很多护栏仍依赖人工执行。

后续可以逐步把高频规则自动化，例如：

- 文档链接和目录入口校验
- API 变更与参考文档同步校验
- Flyway 命名和顺序校验
- 评审模板完整性检查

## 对 ProjectPilot 的落地建议

如果希望继续提升当前仓库的 Harness Engineering 水平，建议按下面顺序推进。

### 第一阶段：先把“能找到”和“能验证”做扎实

- 保持 `AGENTS.md` 作为唯一规则入口
- 继续补齐各目录 `README.md`
- 确保新增文档都能从 `docs/README.md` 或目录索引进入
- 对高频任务补齐检查清单和模板

### 第二阶段：把“完成说明”标准化

- 推广使用 [docs/reviews/templates/verification-evidence-template.md](../reviews/templates/verification-evidence-template.md)
- 推广使用 [docs/plans/task-status-template.md](../plans/task-status-template.md)
- 要求每次关键任务都能留下验证证据
- 让评审结论、验证结论和产出文档相互可追踪

### 第三阶段：把高频护栏自动化

- 把最常被遗漏的规则搬进 CI 或脚本
- 减少只能靠口头提醒才能执行的约束
- 把“人工检查”逐步收敛为“人工判断 + 自动校验”

## 一句话结论

对 ProjectPilot 来说，Harness Engineering 的正确方向不是“把 AI 协作包装成新平台”，而是：

> 用统一规则、清晰导航、外置验证和运行支撑，让人和 AI 在同一个仓库里更稳地完成 Web 产品研发。
