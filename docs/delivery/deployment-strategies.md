# 部署策略

## 目标

本文档定义 HernessDemo 在不同环境中的部署方式、适用条件和回滚策略，避免每次发布临时决定。

## 策略清单

### Recreate

`Recreate` 指先下线旧版本，再上线新版本。

适用场景：

- 低风险内部环境。
- 单实例验证环境。
- 无法并行运行双版本的临时场景。

风险：

- 会造成短暂不可用。
- 不适合生产核心流量。

### Rolling

`Rolling` 指逐批替换旧实例。

适用场景：

- 多实例服务。
- 可以接受小批量替换过程。
- 当前阶段的默认生产候选策略。

优点：

- 资源开销相对较低。
- 不需要完整双套环境。

风险：

- 若版本兼容性差，可能出现新旧实例并存问题。
- 回滚速度通常慢于蓝绿。

### Blue-Green

`Blue-Green` 指保留两套完整运行面，通过流量切换完成发布。

适用场景：

- 对停机容忍度低。
- 需要快速回滚。
- 基础设施允许双套资源。

优点：

- 切换清晰。
- 回滚路径简单。

风险：

- 成本较高。
- 数据库变更需要额外兼容设计。

### Canary

`Canary` 指先让少量流量进入新版本，再逐步扩大放量。

适用场景：

- 用户量较大。
- 关键链路风险高。
- 已具备较成熟的指标监控和流量控制能力。

优点：

- 风险暴露面最小。
- 能结合指标做渐进式验证。

风险：

- 需要更成熟的网关、观测和自动化能力。
- 配置复杂度较高。

## 当前推荐策略

结合当前仓库成熟度，建议采用分阶段策略：

| 环境 | 推荐策略 | 原因 |
| --- | --- | --- |
| dev | Recreate | 简单直接，便于快速联调 |
| test | Recreate 或 Rolling | 取决于测试环境实例规模 |
| staging | Rolling | 开始演练生产节奏 |
| prod | Rolling，后续演进到 Blue-Green 或 Canary | 当前能力更适合先做稳妥演进 |

## 数据库变更策略

数据库迁移与应用发布必须联合设计。

基本原则：

- 优先采用向后兼容迁移。
- 不在同一发布窗口中同时引入不可逆结构破坏和高风险业务改动。
- 先发兼容数据库结构，再切换应用逻辑，再清理旧字段。
- 所有迁移必须通过 Flyway 管理。

## 回滚原则

- 应用版本回滚必须有明确目标版本。
- 回滚前要确认数据库结构是否兼容旧版本。
- 若数据库变更不可逆，必须提前准备补偿方案。
- 回滚动作必须留痕，包括时间、操作者、原因和影响范围。

## 决策规则

选择部署策略时至少评估：

- 服务实例规模。
- 用户影响范围。
- 数据库兼容性。
- 是否具备流量切换能力。
- 是否具备实时指标监控和告警。
- 回滚窗口是否足够短。

## 维护规则

- 生产默认策略变化时，必须同步更新本文档和发布运行手册。
- 新增灰度或蓝绿能力时，必须同步定义流量切换和回滚规则。
