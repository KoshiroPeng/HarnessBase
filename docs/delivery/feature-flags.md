# 特性开关与渐进式发布

## 目标

本文档用于定义 CallCenter 在未来引入特性开关后的治理原则，把“代码发布”和“功能放量”分离，降低高风险功能上线带来的影响面。

## 适用场景

以下场景建议考虑特性开关：

- 风险较高的新业务流程。
- 面向部分租户、组织或角色试点的能力。
- 需要灰度验证的权限、CTI、实时通道、AI 质检或报表能力。
- 前后端需要错峰上线的兼容改动。

## 基本原则

- 特性开关必须有明确负责人和失效时间。
- 特性开关名称必须表达业务语义，禁止使用无意义缩写。
- 重要功能开关必须可观测，至少能记录启用范围和命中量。
- 临时开关不得长期遗留，功能稳定后应尽快清理。

## 开关维度

建议支持以下开关粒度：

- 全局开关。
- 按环境开关。
- 按组织或租户开关。
- 按角色开关。
- 按流量比例开关。

## 当前状态

当前仓库尚未引入专门的特性开关基础设施，本文档先作为治理占位。实际引入时，必须同步补齐：

- 开关存储位置和配置来源。
- 开关变更审计能力。
- 回滚与紧急关闭流程。
- 观测指标和告警。

## 维护规则

- 新增特性开关前必须说明风险、范围和清理计划。
- 涉及权限、安全、CTI 控制、录音和客户敏感数据的开关必须经过额外审查。
