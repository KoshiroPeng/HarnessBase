# 后台代码评审清单

## 评审重点

1. 后台编码规范
2. 异常检查
3. 性能与稳定性风险

## 清单

| 重要性 | 序号 | Checklist |
| --- | --- | --- |
| 必选 | 1 | 变量、常量、方法命名是否符合规范？ |
| 必选 | 2 | 类、方法和关键逻辑是否有必要注释？ |
| 必选 | 3 | 是否对入参、查询结果和公共方法返回值做了合理校验？ |
| 必选 | 4 | 是否对可能出现异常的逻辑进行了合理处理？ |
| 必选 | 5 | 是否存在未定义含义的裸数字？ |
| 必选 | 6 | 是否涉及多线程，是否存在并发安全问题？ |
| 必选 | 7 | 循环是否一定能结束，是否存在高风险多层嵌套循环？ |
| 必选 | 8 | SQL 是否存在高风险性能问题？ |
| 必选 | 9 | 是否存在远程调用，是否考虑了超时和失败处理？ |
| 必选 | 10 | 是否存在单个方法过于繁琐，是否可提取公共逻辑？ |
| 必选 | 11 | 若存在定时任务，是否支持控制启停，是否有重复执行风险？ |
| 必选 | 12 | 若需要处理历史数据，是否已有脚本或迁移方案？ |
| 必选 | 13 | 是否有合理日志输出？ |
| 必选 | 14 | 是否存在频繁操作数据库？ |
| 必选 | 15 | 若使用第三方工具类，是否评估过稳定性问题？ |
| 必选 | 16 | 若存在异步操作，是否考虑数据一致性问题？ |

## 当前项目适配说明

- 代码评审必须同时遵守 [AGENTS.md](../../AGENTS.md) 中的硬性规则。
- 外部调用必须检查是否通过已有 common、api、adapter 或 client 接入。
- 数据库变更必须检查是否已同步 [server/sql](../../server/sql)。
- 后端模块必须遵守 [docs/architecture/boundaries.md](../architecture/boundaries.md) 中的真实微服务边界。
