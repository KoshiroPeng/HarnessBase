# 文档元数据规范

## 目标

本文档用于统一 HernessDemo 内部 Markdown 文档的元数据标头写法，减少文档维护过程中的歧义，提升 AI 与人工协作时的可识别性、可追踪性和可导航性。

## 适用范围

以下文档默认必须使用元数据标头：

- `AGENTS.md`
- `docs/` 下的 Markdown 文档
- `deploy/` 下的 Markdown 文档
- 目录级 `README.md`，前提是该文档承担目录导航、规则索引、场景分流或模板入口职责

以下文档允许豁免：

- 根目录 [README.md](../../README.md)，因为它主要承担对外项目说明和 GitHub 首页入口职责
- 未来若存在纯展示用途、非治理入口的外部说明文档，可按同样原则评估是否豁免

## 必填字段

内部 Markdown 文档的元数据标头必须包含以下字段：

- `last_updated`
- `status`
- `owner`
- `description`

推荐格式如下：

```md
---
last_updated: 2026-06-08
status: active
owner: "@PengKang"
description: 用一句话说明本文档的职责、使用场景或目录入口作用。
---
```

## 字段说明

### `last_updated`

- 表示本文档最近一次有效内容变更日期
- 只要正文、规则、导航、模板字段或适用范围发生变化，就应更新
- 建议使用 `YYYY-MM-DD` 格式

### `status`

- 表示文档当前是否处于有效维护状态
- 当前项目默认使用 `active`
- 若后续需要区分草稿、废弃或归档文档，可再扩展状态值，但应先更新本规范

### `owner`

- 表示文档维护责任人
- 当前仓库统一使用 `@PengKang`
- 若后续文档维护责任发生变化，应同步调整

### `description`

- 用一句话描述文档职责
- 优先说明“这份文档是做什么的”，而不是重复标题
- 对目录级入口文档，应强调其导航、索引或分流作用
- 对模板、清单、规范文档，应强调其检查、记录或约束作用

## 使用建议

- 新增内部 Markdown 文档时，先补齐元数据标头，再写正文。
- 修改带标头文档时，同步更新 `last_updated`。
- 如果一个目录下的 `README.md` 已经承担场景导航作用，应保留元数据标头。
- 如果某份文档只是单纯面向外部展示，不承担内部治理和导航职责，可评估是否豁免元数据标头。

## 与 Harness Engineering 的关系

这套元数据不是为了“把文档写复杂”，而是为了让文档更容易承担 Harness Engineering 中的这些职责：

- 上下文选择
- 项目记忆
- 规则入口识别
- 维护状态追踪

也就是说，元数据标头本质上服务于“更快找到正确文档、判断文档是否有效、知道谁负责维护”。

## 维护规则

- 若新增元数据字段，必须先更新本文档和 [AGENTS.md](../../AGENTS.md)。
- 若调整哪些文档可豁免元数据标头，必须同步更新本文档与 [docs/README.md](../README.md) 中的导航说明。
- 若后续引入自动检查脚本或 CI 校验，应以本文档为规则来源，而不是在脚本里另写一套口径。
