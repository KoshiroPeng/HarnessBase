# 发布治理与权限模型

## 目标

本文档定义 HernessDemo 的发布权限、审批要求和审计原则，确保高风险环境变更可控、可追溯。

## 角色建议

### 开发负责人

职责：

- 提交代码并修复质量问题。
- 补充测试和相关文档。
- 提供变更说明、风险点和验证建议。

### 发布负责人

职责：

- 确认发布窗口和环境目标。
- 核对版本、配置和依赖变更。
- 组织发布执行与验证。

### 审批人

职责：

- 对高风险环境或高风险变更进行审批。
- 关注影响范围、回滚路径和资源准备情况。

### 值班人

职责：

- 在观察窗口内关注告警和异常。
- 必要时协助执行回滚和升级处理。

## 权限原则

- 开发环境可相对开放，但仍需保留变更记录。
- test 和 staging 的部署权限应收敛到明确角色。
- prod 的部署、回滚和密钥变更必须受控。
- 关键环境操作必须可审计到人、时间、版本和结果。

## 审批建议

以下情况建议要求额外审批：

- 生产环境发布。
- 涉及数据库结构变更。
- 涉及认证、权限、计费和支付逻辑变更。
- 涉及密钥轮换或基础设施调整。
- 涉及高风险回滚操作。

## 审计要求

至少记录以下内容：

- 发布版本。
- 目标环境。
- 操作者。
- 审批人。
- 发布时间。
- 验证结果。
- 回滚结果与原因。

## 当前最小落地建议

1. 先为 `staging` 和 `prod` 明确发布负责人。
2. 把生产发布审批条件写入流程文档。
3. 在 CI/CD 中保留构建、部署和验证日志。
4. 为回滚操作保留单独审计记录。

## 当前 GitHub 对应入口

当前仓库中的对应入口包括：

- `.github/workflows/server-release.yml`
- `docs/delivery/github-environments.md`
- `deploy/release/release-checklist.md`

建议通过 GitHub Environments 实现最小权限与审批落地，通过发布检查清单统一审批前后的人工核对动作。

## 维护规则

- 新增高风险模块时，必须同步评估是否需要调整审批范围。
- 组织角色变化时，必须及时更新本文档。
